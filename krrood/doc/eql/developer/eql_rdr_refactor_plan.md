# EQL-Based Ripple Down Rules — Refactoring Plan

## Goal

Refactor the Ripple Down Rules (RDR) package so the rule tree is a native **EQL
(Entity Query Language) expression tree** instead of generated nested
`if/elif/else` Python functions reconstructed via AST.

Start minimally: a `SingleClassRDR` over the **zoo dataset**, predicting an
animal's `species` (the underspecified attribute).

### Hard constraints (from the user)

1. **Evaluation is pure EQL evaluation.** No separate recursive SCRDR loop —
   `Refinement` / `Alternative` / `Add` already encode SCRDR semantics, the EQL
   engine does the traversal.
2. **No strings, ever.** The rule tree is held in memory only as live EQL
   `SymbolicExpression` objects. Persistence is **Python files** (generated by
   unparsing the live DAG; imported back to rebuild it). No JSON, no
   `conditions: str` fields, no AST round-trip, no `CallableExpression` code
   strings.
3. **Expert answers are live EQL expression objects**, not lists or strings.
4. **The conclusion is known during fit** (ground truth exists for the zoo
   case), so the expert is asked **only for a conditions expression**; the RDR
   builds the `Add(animal.species, target)` conclusion itself.
5. `Animal` needs **no special EQL treatment**. The IPython shell already
   captures the caller scope; only the EQL factories are missing from it.
6. The `RDRDecorator` path is **out of scope** for this refactor (future phase).

## Architectural decision: RDR as an AOP observer on EQL evaluation

The `inference_explanation` branch (merged into `rdr_refactor`) provides the
aspect-oriented hooks the RDR needs:

- `entity_query_language/evaluation.py` — `EvaluationObserver` (ABC, hooks:
  `on_evaluate_enter`, `on_evaluate_exit`, `on_result_yielded`,
  `on_conclusions_processed`) and `EvaluationContext` carried in a `ContextVar`.
- `core/base_expressions.py::SymbolicExpression._evaluate_` fires those hooks at
  every node. If an `EvaluationContext` is **already set** before evaluation
  starts, it is reused — so an external controller can install its own observers
  before calling `evaluate()`.
- `_replace_child_` enables live DAG rewiring (already used by
  `create_and_update_rule_tree`).
- `rules/conclusion_selector.py` — `Refinement` (except-if = SCRDR refinement
  edge), `Alternative` (else-if = SCRDR alternative edge), `Next` (union, MCRDR).
- `rules/conclusion.py::Add` — the conclusion primitive (sets a binding).
- `_monitoring.py::monitored` — decorates `Variable` and `Query`; captures the
  `_creation_stack: CallStack` of where an EQL object was created.
- `explanation/explanation.py::InferenceRecorder` — reference example of an
  observer that records *why* an instance was inferred. The RDR observer mirrors
  its shape.

The RDR is therefore an **orthogonal controller**: the EQL DAG is the domain
object, the RDR observes evaluation, checks the inferred conclusion against
ground truth, consults the expert, and rewires the DAG.

### Note: evaluation API after the merge

`_evaluate__(self, sources: OperationResult)` — single `OperationResult` source
argument (truth pulled, not pushed). No separate `parent`/`Bindings` argument.
All new code targets this signature.

## Two reuse decisions

### Insertion point — reuse observer hooks, no chain-walking

- `on_conclusions_processed(expression, result)` fires at the conditions-root
  that produced a conclusion, after its `Add` ran. The observer is handed the
  **firing node directly** — that node is the refinement anchor when the
  conclusion is wrong.
- `result.satisfied_condition_ids` (`OrderedSet[UUID]`, populated by the default
  `SatisfiedConditionTracker`) + `query_root._get_expression_by_id_(id)` give the
  full set of satisfied condition nodes for depth selection / expert context.

Rule:
- Conclusion fired but wrong → anchor from the hook → `Refinement.insert_at(anchor, new_conditions)`.
- Nothing fired for `species` → `Alternative.insert_at(root_conditions, new_conditions)`.

### Scope capture — reuse the `@monitored` creation stack

`@monitored` decorates `Variable` (`core/variable.py`) and `Query`
(`query/query.py`); on construction it captures `_creation_stack` via
`inspect.stack()`, and `CallStack.filter()` drops site-packages so the outermost
*user* frame (where `underspecified(Animal, ...)` was called) is identifiable.

Gap: `StackFrame.from_frame_info` drops `f_locals`/`f_globals` to avoid leaks.
Fix: for non-site-packages frames, retain a shallow snapshot of
`{**f_globals, **f_locals}`. A helper `get_definition_scope(variable_or_query)`
returns the merged caller scope; the IPython shell scope = that snapshot + EQL
factories. This replaces the old `CaseQuery.scope` `f_back.f_back.f_globals` hack
and brings `Animal` and everything else in the caller namespace along for free.

## Phases

Dependency order: **Phase 0 (scope capture) and Phase 6 (unparser) are the two
net-new EQL-side capabilities**; everything else composes merged primitives.
Each phase is tested well before moving on.

### Phase 0 — Scope capture (EQL side, foundational)
Extend `StackFrame`/`monitored` to optionally snapshot `{**f_globals, **f_locals}`
for user frames. Add `get_definition_scope(variable_or_query)` returning the
merged caller scope. Add the EQL factories (`entity`, `an`, `variable`,
`Refinement`, `Alternative`, `Add`, comparators) into that scope.

### Phase 1 — Zoo dataset as plain `Animal` instances
`Animal` dataclass (17 zoo fields + `species: Optional[Species] = None`) and a
loader returning `List[Animal]`. No `Case`. Shared `animal = variable(Animal,
domain=animals)` created once.

### Phase 2 — `RDRObserver(EvaluationObserver)`
Modeled on `InferenceRecorder`. Hook `on_conclusions_processed` to capture the
firing conditions-root node + inferred `species` binding. RDR installs
`EvaluationContext(observers=[rdr_observer, EvaluationTracker(),
SatisfiedConditionTracker()])` before `entity(animal).evaluate()`.
Classification = pure EQL evaluation.

### Phase 3 — Live-DAG rule growth
Add stack-free `Refinement.insert_at(anchor, new_conditions)` /
`Alternative.insert_at(anchor, new_conditions)` on `ConclusionSelector`, built on
`_replace_child_`. Anchor from the observer hook; `satisfied_condition_ids` for
depth.

### Phase 4 — Expert interface (conditions only)
Reuse the IPython shell with the Phase-0 captured scope + EQL factories. Expert
`return`s a live EQL conditions expression. RDR builds
`Add(animal.species, target_species)` and wires it under the new conditions.

### Phase 5 — `EQLSingleClassRDR` orchestration
- `classify(animal)`: install context, evaluate, read `species` binding.
- `fit_case(animal, target_species)`: classify; if wrong/missing, prompt expert
  for conditions only, insert rule live, re-evaluate.
- Underspecified entry: `underspecified(Animal, ...)` /
  `Match(Animal)(..., species=Match(Species))`.

### Phase 6 — Persistence (Python files only; the unparser)
`EQLRuleTreeSerializer`: traverse the live DAG, emit a `.py` module of
`with Refinement(...) / Alternative(...): Add(...)` blocks
(`Attribute`→`animal.hair`, `Comparator`→`a == b`, ...). Load = `import`.
Verified there is **no** existing EQL→Python source emitter (repr/verbalization
are display/NL only), so this is genuinely new.

### Phase 7 — Underspecified backend + tests
The RDR is wired up as a **backend for underspecified (`...`) queries**, the same
`Match`/ellipsis pattern the probabilistic backend consumes
(`test/krrood_test/test_underspecified_knowledge/`).

- `underspecified(type, domain=..., target_type=...)` now optionally carries a
  domain (the `domain` field was hoisted from `MatchVariable` up to `Match`).
- `rdr/underspecified.py::UnderspecifiedMatch` reads an underspecified `Match`:
  finds the `...` inference targets, guards single-class to single-valued
  attributes (rejects unbounded iterables → future `MultiClassRDR`), strips the
  ellipsis conditions, and lazily streams the domain instances passing the filter.
- `rdr/backend.py::RDRBackend` is the **lazy filter-then-infer pipeline** (decision
  (1), not the unified single-DAG): one `EQLSingleClassRDR` per `(case type,
  attribute)`; `infer()` yields `UnificationDict`s by default (read via
  `query.variable` / `query.variable.<attr>`) or fills instances in place
  (`fill_in_place=True`); auto fit-mode when no model exists.
- Fit mode is ground-truth-aware: with a target the expert gives conditions only;
  without one the expert gives both (`Expert.ask_for_rule`). `EQLSingleClassRDR`
  supports `from_underspecified(template)` and `fit_case(target=None)`.
- Generative inference (`inference(Drawer)(...)` — invent a new instance) is out of
  scope for this phase.
- Serializer fix: `and_`/`or_` rule conditions now serialize (conclusions attached
  as connective children are excluded from the emitted boolean expression).

Tests: `test_zoo_loader`, `test_observer`, `test_rule_tree_growth`,
`test_single_class_rdr`, `test_interactive_expert`, `test_serialization`,
`test_underspecified_rdr` (target discovery, ellipsis stripping, scalar guard, lazy
infer, fill modes, fit modes, fit→save→load→infer).

### Phase 8 — Expert / ExpertInterface split + interactive ergonomics

Phase 4 collapsed *what* to interact/ask/retrieve and *how* to interact/ask/retrieve into a single
`IPythonExpert(Expert)`. Phase 8 separates them and ports the ergonomics of the
legacy `ripple_down_rules` IPython shell.

#### The Strategy split (composition, not inheritance)

- **`Expert`** (`rdr/expert.py`) — concrete `@dataclass` holding a field
  `interface: ExpertInterface`. It owns *policy*: conditions-only when the target
  is known, conclusion+conditions when not (`ask_for_conditions` / `ask_for_rule`,
  signatures unchanged so `single_class.py` call sites are untouched). It also owns
  the **validators**: conditions must be a live `SymbolicExpression` (a plain
  `bool` means the expert wrote over the concrete case — re-prompt); conclusion
  must be non-`None`.
- **`ExpertInterface`** (`rdr/interface.py`) — abstract `@dataclass`. It owns
  the *Interaction Mechanism*: a template-method `interact(context, requests) -> dict[str, Any]`
  running the build-namespace → render → run → validate → re-prompt loop. Concrete
  interfaces implement only `_run(namespace, header)`.
- **`IPythonInterface`** (`rdr/interactive.py`) — the embedded-shell implementation.
  Keeps an injectable `shell_runner` seam so tests can simulate the human without a
  terminal. `IPythonExpert` is **removed**; callers use
  `Expert(interface=IPythonInterface())`.
- **`FunctionInterface`** (`rdr/interface.py`) — a programmatic, non-interactive
  interface wrapping `(context, requests) -> dict`. The reusable test double that
  replaces the old pattern of subclassing the `Expert` ABC.

Value objects: `CaseContext` (`case_instance`, `case_variable`,
`current_conclusion`, `target_conclusion` (sentinel-absent for `ask_for_rule`),
`scope`) and `AnswerRequest` (`name`, `validate: Callable[[Any], Optional[str]]`
returning an error message or `None`, `example`, `required`).

#### Shell namespace naming

- `case_instance` — the concrete object (experiment freely: `case_instance.milk`
  → `True`; native Jedi completion since it is a real instance).
- `case_variable` — the shared EQL variable wired into the rule-tree DAG; conditions
  must be built over it (`conditions = case_variable.milk == True` → live
  `SymbolicExpression`). It is injected as the **same object** in the tree, so it
  cannot be rebuilt via the `variable(...)` factory inside the shell.
- Plus the EQL factories via `get_definition_scope(case_variable)`, and each answer
  name pre-seeded to `None`.

#### Error-tolerant loop, colour, table

- `interact` re-renders a coloured error and reopens the shell on missing/invalid
  answers (e.g. `case_instance.x == True` yields a `bool`, caught by the
  `is SymbolicExpression` validator with a "build it over `case_variable`" hint).
  An **explicit exit** (`exit`/`quit`/Ctrl-D) aborts → raises
  `NoConditionsProvided` / `NoConclusionProvided`.
- Coloured instruction header via `colorama`; case rendered as a table via
  `rdr/case_table.py::render_case_table` (`tabulate`, terminal-width aware,
  lowercased bools, truncation). No import from the legacy `ripple_down_rules`
  package (its `__init__` spins up a QApplication).

#### Autocomplete — core `__dir__`

`CanBehaveLikeAVariable.__dir__` returns its real members unioned with the public
attribute names of `self._type_`, read via `self.__dict__` (never `getattr`, which
would route through `__getattr__` and never return `None`). `__getattr__` (every
name → `MappedVariable`) is **wholly untouched**; `__dir__` only changes what
completion offers. Benefits every interactive EQL variable, so `case_variable.<tab>`
now offers the case type's attributes.
