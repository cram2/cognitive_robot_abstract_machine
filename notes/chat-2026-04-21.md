## Assistant
Within this framework, I would treat the `World` as the source of truth for perceived state, and add a small rule layer on top of it. I would not put much logic into `Context` itself, because `Context` is intentionally thin: it just carries `world`, `robot`, and execution flags. The better place is:

1. Perception updates `world`
2. Semantic annotations describe surfaces and objects
3. An event evaluator inspects `world` and emits tasks/events

That fits the demo in `demo.py`, where the world is built, semantic annotations are added, and reasoning is run before planning.

A clean shape for it would be a separate dataclass like this:

```python
from dataclasses import dataclass, field
from datetime import datetime
from typing import Callable

from semantic_digital_twin.semantic_annotations.semantic_annotations import (
    Table, Fridge, Milk
)

@dataclass
class WorldEvent:
    name: str
    payload: dict

@dataclass
class WorldRule:
    name: str
    condition: Callable[["WorldEventEngine"], bool]
    build_event: Callable[["WorldEventEngine"], WorldEvent]
    once: bool = False

@dataclass
class WorldEventEngine:
    world: object
    now_fn: Callable[[], datetime]
    emitted: set[str] = field(default_factory=set)

    def objects_on_surface(self, surface_annotation):
        return list(getattr(surface_annotation, "objects", []))

    def all_tables(self):
        return self.world.get_semantic_annotations_by_type(Table)

    def all_fridges(self):
        return self.world.get_semantic_annotations_by_type(Fridge)

    def has_milk_on_table(self) -> bool:
        for table in self.all_tables():
            if any(isinstance(obj, Milk) for obj in self.objects_on_surface(table)):
                return True
        return False

    def evaluate(self, rules: list[WorldRule]) -> list[WorldEvent]:
        events = []
        for rule in rules:
            if rule.once and rule.name in self.emitted:
                continue
            if rule.condition(self):
                events.append(rule.build_event(self))
                if rule.once:
                    self.emitted.add(rule.name)
        return events
```

Then define rules such as:

```python

        condition=lambda e: e.now_fn().hour >= 10 and any(

        ),
        build_event=lambda e: WorldEvent(

            payload={"reason": "it is 10:00 and there are objects on tables"}
        ),
        once=True,
    ),
    WorldRule(
        name="milk_should_go_to_fridge",
        condition=lambda e: e.has_milk_on_table(),
        build_event=lambda e: WorldEvent(
            name="store_in_fridge",
            payload={"object_type": "Milk"}
        ),
    ),
]
```

The important part is how the world gets populated while driving. I would do it in a repeated loop over navigation waypoints, not as one giant `NavigateAction` with hidden side effects:

```python
for waypoint in patrol_waypoints:
    NavigateAction(waypoint).resolve().perform()

    with world.modify_world():
        detected_objects = detect_visible_objects_somehow(world, robot)
        update_surface_assignments(world, detected_objects)

    events = event_engine.evaluate(rules)
    for event in events:
        dispatch_event(event)
```

The core model is:

- `world` stores bodies, poses, semantic annotations
- surface annotations such as `Table` come from `semantic_digital_twin.semantic_annotations.semantic_annotations`
- supporting surfaces are already a first-class concept through `HasSupportingSurface`
- objects can be attached to a surface/container using the storage-space pattern in that same mixin stack

So for your apartment example, I would represent “table”, “countertop”, “fridge”, and “milk” as semantic annotations inside `world`, and keep an event engine that derives higher-level meaning from them.

The missing piece is usually “how do I know an object is on a table/countertop?” There are two good options:

- Explicit attachment: when perception decides an object belongs to a surface, add it to that surface annotation’s `objects` list via the storage/support API. This is best if you want symbolic persistence.
- Spatial query: compute “object pose is inside or above supporting surface region” each cycle. This is best if you want purely geometric inference.

In this framework, I would prefer explicit attachment after perception, because then your rules become simple and symbolic:

```python
for table in world.get_semantic_annotations_by_type(Table):
    if any(isinstance(obj, Milk) for obj in table.objects):
        emit("store_in_fridge")
```

So the short answer is: keep the event logic in a separate dataclass that reads the `World`; use `world.modify_world()` to update semantic state during patrol; and model tables/countertops as `HasSupportingSurface` annotations with attached perceived objects. That gives you robust triggers like “clean up at 10am” and “milk belongs in the fridge” without overloading the base `World` or `Context` types.

## Assistant
`emit(...)` in the example was not a built-in function from this framework. It was shorthand for “create or publish an event when a rule matches.”

So this line:

```python
emit("store_in_fridge")
```

just means: “the world-state checker detected a situation, and now we notify the rest of the system.”

In practice you would replace it with one of these patterns:

1. Return an event object

```python
events.append(
    WorldEvent(
        name="store_in_fridge",
        payload={"object": milk_annotation}
    )
)
```

2. Push into an in-memory queue

```python
self.pending_events.append(
    WorldEvent(name="store_in_fridge", payload={"object": milk_annotation})
)
```

3. Directly trigger a plan/action

```python
TransportAction(
    milk_body,
    fridge_target_pose,
    Arms.LEFT,
).resolve().perform()
```

4. Publish to ROS or another external event bus

Inside this codebase, the simplest interpretation is usually option 1 or 2: evaluate the `world`, produce `WorldEvent` objects, then a dispatcher decides what plan to run.

For example:

```python
@dataclass
class WorldEvent:
    name: str
    payload: dict

def dispatch_event(event: WorldEvent, context: Context):
    if event.name == "store_in_fridge":
        milk = event.payload["object"]
        fridge_pose = event.payload["target_pose"]

        TransportAction(
            milk.root,
            fridge_pose,
            Arms.LEFT,
        ).resolve().perform()
```

So:

- `emit` = “signal that a condition became true”
- it is not part of `World`
- it is not part of `Context`
- you need to define what happens when an event is emitted

A better explicit version than `emit(...)` would have been:

```python
return WorldEvent(name="store_in_fridge", payload={...})
```

because that makes it obvious that the rule engine is only detecting situations, not executing robot behavior yet.
